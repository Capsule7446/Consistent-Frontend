# 一致性:Design System + 状态机(定业务语义)

> `consistent-frontend` 参考之三。哲学定方向、结构定落点之后,这一层回答:**业务怎么迭代,UI/UX 与代码怎么保持强一致、不漂移?** 用设计系统锁住"长什么样",用状态机锁住"能做什么"。
> 语言/框架无关;示例用 TS 表意。

## 它挡的是哪种腐化

哲学挡后端形状、结构挡代码乱放;这一层挡**业务语义漂移**:

- 同一操作(作废、提交、审批、批量、筛选)在不同页面长出不同 UI、不同代码 → 体验割裂,改一处规则追十处实现。
- 业务规则迭代了,只改了部分页面 → 业务逻辑与前端实现"对不上账",越迭代越散。

根因:没有把"业务语义"沉淀成**可复用的契约**。解法是两件单一真相:**设计系统**(管长相)+ **状态机/状态→操作**(管行为)。

## 三件固化物

### 1. 交互通用语言 —— 一份词汇表,全端同词

业务名词/动词在 VM、组件名、路由、文案里**用同一个词**(沿用后端通用语言,见 `philosophy-ddd-clean.md`)。

```text
作废 void   —— 使已生效单据失效,不可逆,留痕。≠ 删除(物理移除)≠ 取消(下单前放弃)
退款 refund —— 已支付订单原路退回,可部分。
```

不能这页"取消"、那页"删除"、代码里 `cancel`。有歧义回领域建模对齐,不私造词。

### 2. 操作 → 模式 映射表 ★ 核心(落在 `shared/ui`)

把业务操作**归类**,每类绑定**唯一**的交互模式 + **唯一**的设计系统组件契约。任何同类操作必须复用——这是设计系统"模式库 / 设计语言"在业务侧的延伸,也是前端版"聚合不变量:同种命令处处一致"。

| 操作类型 | 业务语义 | 固化的 UI 模式 | 设计系统契约(框架无关) |
| :--- | :--- | :--- | :--- |
| 破坏性 | 不可逆/有副作用(作废、删除、停用) | 二次确认 + **具名影响** + 可撤销窗口 | `ConfirmAction({ impact, danger, onConfirm })` |
| 异步提交 | 写操作(创建、保存、提交) | 乐观更新 + 失败回滚 + 结果反馈 + 防重复 | `SubmitAction({ mutation })` |
| 列表筛选 | 缩小集合 | URL 同步工具栏 + 计数 + 空态 | `FilterBar({ schema, value })` |
| 多步流程 | 向导/审批链 | 步骤机 + 草稿态 + 可返回不丢数据 | 状态机驱动(见下) |
| 权限/状态门控 | 角色或状态不允许 | **统一禁用 + 原因提示**,而非默默隐藏 | `Gated({ allowed, reason })` |
| 批量 | 对多选执行同一命令 | 选择态 + 影响计数 + 部分失败可见 | `BulkAction({ selection, command })` |

要点:**表是固化物,不是建议**。新操作先**归类→复用**;模式确实不够用,**评审后新增一行**,而不是某页私自发明第七种确认弹窗。一个类型 = 一种体验 = 一份契约。

### 3. 状态 → 操作 不变量 + 状态机(落在 `entities/model`、`features/model`)

**静态可用性**用"状态→操作"声明:每个实体的状态 → 允许操作集中一处,UI **从这里推导**,而不是各页 `if (status === 'PAID')`。

```ts
// entities/order/model/actions.ts —— 单一真相
export const orderActions = {
  DRAFT:    ["submit", "edit", "void"],
  PAID:     ["refund", "void"],
  REFUNDED: [],
} as const;
// UI: const allowed = orderActions[order.status]; 决定按钮显隐/禁用,组件无 status 硬编码
```

**动态流程**(多步/异步/带重试)用**状态机**显式建模——这才是前端真正"可测试的业务逻辑",从渲染里抽出来。

```ts
// features/void-order/model/machine.ts(XState 表意,框架无关)
// idle → confirming → submitting → (done | error→ retry/cancel)
```

> 规则一变("已退款也能再开票")→ 只改 `orderActions` / 状态机这一处 → 列表页、详情页、批量栏**同时**更新,不漏页。

## 设计系统 / Design Tokens(落在 `shared/ui` + `shared/config`)

- 第三方 UI 库**只在 `shared/ui` 被引用**,包成自有 `<Button>/<Table>/<ConfirmAction>`;换库/升级只动 `shared/ui`。
- **设计令牌**(颜色、间距、字号、圆角)集中在 `shared/config`/`shared/ui`,组件不写魔法值 → 视觉一致且可整体换肤。
- 操作→模式映射表里的契约组件,就建在这里,是"长相一致"的唯一真相。

## 迭代一致性:怎么保证"改一次,处处对"

1. **改在源头,不在页面打补丁**。规则变 → 改词汇表 / 操作→模式表 / 状态→操作表三者之一。
2. **由表生成,不手抄**。按钮可用性、文案、确认流程从三张表推导;页面只**声明**"这是个『破坏性操作』、作用于『订单』"。
3. **一致性门禁**(lint/审查):同类操作不得绕过设计系统契约自造实现;实体可用性不得在组件里散落判断。

## 固化为约束

- lint:业务层不直引第三方 UI 库(只能引 `shared/ui`)。
- lint:`components`/展示层禁止对实体 `status` 硬编码分支(必须取自状态→操作声明)。
- lint:禁止业务代码自拼确认弹窗(破坏性操作必须走 `ConfirmAction`)。
- 约定:每个实体一份状态→操作声明;每类新操作进映射表才能落地;词汇表随 PR 评审。
