# Workflow:Consistent Frontend(强一致前端开发)

> 把"三叠加"的开发决策串成可门禁、可回溯的流程。由 `/consistent-frontend:fe-build` 触发。
> 目标:让代码改起来便宜、让业务与前端不脱节。**DDD/Clean 定方向、FSD 定结构、Design System+状态机 定语义**——三者各挡一种腐化。

## 链路

```
consistent-frontend
   → ⓪ 对齐语义 ★      （通用语言 + 操作→模式映射 + 状态→操作;地基,贯穿）
   → ① 定切片与契约     （slice 落哪层 + VM/数据入口签名/props;用通用语言命名）
   → ② 建 entities ★    （model: VM + 状态→操作 | api: 请求→schema→mapper→错误翻译,DTO 不外泄）
   → ③ 建 features      （用例编排 + 状态机;无 UI 业务规则）
   → ④ 建 UI            （shared/ui 设计系统封装 | 展示组件纯 props,操作走契约,可用性由声明推导）
   → ⑤ 装配            （pages 装配 + 路由收口;app 组合根/DI;无业务逻辑）
   → ⑥ 固化            （import 规则 lint + 一致性 lint,入 CI）
```

决策**由外向内**(先收口语义与契约),实现**由内向外**(领域先稳,UI 后接)。
迭代时从 ⓪ 起步:业务规则一变,**先更新三张表,再改下游**——这是"业务与前端不脱节"的纪律。
`--mode retrofit` 时,②~⑤ 以**切片**为单位:每片先补特征化测试锁现状,再按契约解耦替换,旧路径全程可用。

## 阶段门禁(gate)

| 门禁 | 位置 | 放行条件 |
| :--- | :--- | :--- |
| G0 语义门禁 | ⓪ 后 | 用词进通用语言;操作已归类到映射表(同类复用);涉及实体有状态→操作声明 |
| G1 契约门禁 | ① 后 | slice/层归位明确;VM/数据入口/props 已定且稳定,可并行/可 mock |
| G2 领域门禁 | ② 后 | 数据只经一个数据入口进入;DTO 只在 entities/api;边界有 schema 校验与错误翻译 |
| G3 用例门禁 | ③ 后 | 交互逻辑在状态机/用例里,未渗进渲染;features 不互引 |
| G4 一致性门禁 | ④ 后 | 展示组件纯 props;操作走设计系统统一契约;可用性取自状态→操作;第三方 UI 仅在 shared/ui |
| G5 装配门禁 | ⑤ 后 | pages 只装配,无业务逻辑/取数内联;路由在页面收口 |
| G6 固化门禁 | ⑥ 后 | import 规则 + ≥1 条一致性约束已固化为 lint,入 CI |
| G7 切片安全门禁 | 每切片(retrofit) | 现有行为已被特征化测试覆盖,且有回滚点 |

## 回溯矩阵

| 触发条件 | 在哪发现 | 回退到 |
| :--- | :--- | :--- |
| 操作归不进任何模式 / 同操作多种措辞 | ⓪ 语义 | DDD `ddd-contexts` 补通用语言;评审后再定是否新增模式 |
| 业务规则改了只动了部分页面 | 任意 | 回 ⓪ 改三张表,由表推导,全端同步 |
| slice 互相想直引 | ② / ③ | 边界划错或该上提:把组合上移到 widget/page,不破坏 import 规则 |
| 组件还在直接 fetch / 读全局 / 读路由 | ④ | 回 ② 把取数收进数据入口;路由参数在 page 读后下传 |
| 后端 DTO 渗进组件 | ④ | 回 ② 在 entities/api 用 mapper/schema 挡住 |
| 同类操作各页长得不一样 | ④ / 审查 | 回 ⓪ 归类到统一模式,组件改走 shared/ui 契约 |
| 页面越写越胖 | ⑤ | 逻辑下沉到 features/状态机,页面只留装配 |
| 某业务规则在前端被重复实现 | 任意 | 移回后端(单一真相),前端只做即时校验 |

## 与 DDD 体系的衔接

> `ddd-contexts`、`workflow-brownfield` 属于**另一个 DDD 插件**;跨插件引用须加该插件前缀(如 `<ddd-plugin>:ddd-contexts`),本插件不内置它们。

- **同语言**:前端交互通用语言**沿用**后端领域通用语言(`ddd-contexts`),不另起炉灶;前端是独立"呈现上下文"。
- **同建模**:操作→模式映射 ≈ 领域命令;状态→操作声明/状态机 ≈ 聚合不变量在表现层的投影;`entities/api` 的 mapper/schema 就是反腐层(ACL)。
- **同纪律**:retrofit 的切片化 + 特征化测试,与 DDD `workflow-brownfield` 一致。
- **结果而非目的**:照此开发,反腐化(不被后端/UI库/路由干扰)与 UI/UX 强一致是自然达成的副产品。

## 编排纪律

- **语义先行**:没对齐通用语言与操作模式不写契约——这是"业务与前端一致"的源头。
- **契约先行**:没定稳定契约不写实现。
- **方向不可破**:依赖只向内、同层不互引、走 public API;用 lint 强制,不靠自觉。
- **改在源头**:迭代先改三张表,由表推导下游,严禁页面打补丁。
- **框架是机制不是结构**:换框架只换实现机制(hook/composable/store),结构/ACL/状态机/操作契约不变。
- **一次只动一片**(retrofit):完成、验收、可回滚后再取下一片。
